# Points 上线前现状评估与行动计划

## 当前实现概览
- **单页集中式 UI**：`src/App.tsx` 体量巨大，直接在组件内部构造学生、任务、成就、社交等所有模拟数据，界面基本覆盖需求但缺乏持久化能力。
- **服务层已定义但未落地**：`src/services/` 下封装了任务、积分、成就、认证等 REST 接口；绝大多数前端组件仍直接依赖本地 mock，而未调用这些服务。
- **认证上下文待接入**：`AuthProvider` 负责 Token 缓存与用户刷新，但应用中缺失登录/注册界面，无法进入真实的家长/学生会话流程。
- **测试覆盖欠缺**：仅有 `RewardsSection` 的回退逻辑单测，其余核心流程（任务完成、积分结算、兑换审批）均未验证。

## 上线阻碍与风险
- **角色切换存在运行时错误**：`currentUser` 使用 `const [currentUser] = useState('student')`，切换按钮会触发未定义的 `setCurrentUser`，家长端完全不可达。
- **任务日历 API 未实现**：`TaskPlanning` 调用 `taskService.getTasksByDateRange`，服务层并无对应方法，组件初始化即报错，无法展示真实日程。
- **奖励与审批仍为占位实现**：`RewardsSection` 依赖 `/api/rewards/available` 占位接口并回退本地数据，兑换流程无法和家长审批、余额扣减串联。
- **关键业务数据仅存于内存**：任务完成、积分变动、兑换请求等逻辑都在 `App.tsx` 内直接改写 `useState`，刷新即丢失，也无法驱动服务层的统计接口。
- **类型检查被绕过**：多个核心模块用 `@ts-nocheck` 避免 TS 报错，大型组件难以维护；缺乏模块拆分也增加重构复杂度。
- **核心闭环验证不足**：`CoreFlowValidator` 仅检查 `localStorage` 标记，无法证明“做任务→拿积分→兑换奖励”闭环真实可行。

## 建议的优先级路线
1. **修复阻塞性缺陷**：恢复角色切换的状态管理、补齐任务服务缺失方法、为奖励列表接入真实接口（或明确 mock 策略并显式提示）。
2. **打通核心流程**：从登录开始串联任务调度、完成确认、积分结算、奖励兑换、家长审批整个链路，所有状态以 API 为准并支持失败重试。
3. **重构状态管理**：将 `App` 拆分为学生端、家长端等路由模块，引入 React Query/Zustand 等方案集中处理数据获取与缓存，逐步移除 `@ts-nocheck`。
4. **完善认证体验**：提供登录/注册/邀请码流程，让 `AuthContext` 真正加载家庭成员信息并决定初始导航。
5. **补齐家长控制台**：将当前的审批、积分调整逻辑改为调用 `pointsService` 与 `taskService`，确保家长能够实际操作生产数据。
6. **扩展测试与监控**：为任务、积分、兑换等关键路径编写单测/集成测试，必要时引入 MSW 模拟 API；上线前准备基础日志与告警机制。

## 质量保障建议
- 引入端到端 smoke 测试覆盖“做任务→积分到账→兑换→审批”两条角色路径。
- 补充 API 合同测试或 Mock Service Worker，提前暴露接口契约变更。
- 在 Docker/Nginx 部署中配置正确的 `VITE_API_BASE_URL` 与安全 headers，并规划备份策略以防数据丢失。

## 下一阶段任务分解
- **Sprint 1（稳定性）**：修复角色切换、补足任务/奖励服务缺口，确保页面不再报错；整理 mock 数据与实际 API 的切换机制。
- **Sprint 2（闭环落地）**：实现登录→任务→积分→兑换闭环及家长审批，所有副作用通过服务层完成。
- **Sprint 3（架构与类型）**：拆分 `App.tsx`、恢复 TypeScript 校验、引入共享状态管理方案，并为关键模块补充测试。
- **Sprint 4（家长控制与监控）**：完成家长仪表盘真实操作、添加日志和基础监控、准备上线验收 checklist。

> 建议在进入下一轮开发之前，先确认后端接口的可用性与数据模型，避免前端重构后仍需大幅调整。
